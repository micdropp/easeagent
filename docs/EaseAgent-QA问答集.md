## 一、架构设计类

### Q1: 纯AI托管的智能办公室方案可行吗

**结论：可行，但初版方案(v1)不够格叫"AI Agent"。**

v1 方案的决策逻辑本质上是硬编码规则：

```
IF 检测到有人 THEN 开灯
IF 无人超过15分钟 THEN 关空调
```

真正的 AI Agent 应该是：

```
看到办公室场景 → 理解当前情况 → 结合员工偏好 → 自主推理决策 → 执行动作
```

因此 v2 方案引入了多模态大模型作为决策核心，用 Function Calling 替代 IF-ELSE 规则引擎。

---

### Q2: 整体架构五层分别做什么

| 层级 | 名称 | 比喻 | 职责 |
|------|------|------|------|
| 第1层 | 硬件设备层 | 四肢 | 摄像头、灯光、空调、屏幕、传感器等物理设备 |
| 第2层 | IoT设备层 | 神经网络 | MQTT Broker 连接硬件和软件，RTSP 管理摄像头，传感器采集 |
| 第3层 | AI处理层 | 眼睛 | YOLO 人员检测、InsightFace 人脸识别、多模态大模型场景理解 |
| 第4层 | 核心服务层 | 大脑 | Agent 决策引擎、事件总线、定时任务、WebSocket 实时通信 |
| 第5层 | 用户层 | 交互界面 | 飞书小程序(面向员工)、Web 管理面板(面向管理员) |

v2 架构将 AI 层进一步拆分为**反射层**(YOLO，毫秒级)和**认知层**(大模型，秒级)，新增了**记忆层**(SQLite+ChromaDB+Redis)。

---

### Q3: 为什么不用多模态大模型而只用YOLO

**这个问题指出了 v1 方案的核心不足。**

只有 YOLO 的系统：
```
传感器信号 → IF-ELSE规则 → 控制指令（不是真正的AI Agent）
```

引入多模态大模型后：
```
多模态感知 → 大模型理解推理 → 工具调用(控制设备) → 反馈学习（真正的AI Agent）
```

**最终方案采用双层架构**：
- YOLO 做"反射"（每秒检测，毫秒级响应，处理简单的有人/无人）
- 多模态大模型做"思考"（事件触发，秒级响应，处理场景理解和复杂决策）

推荐的本地模型：**Qwen2.5-VL-7B**（开源、中文好、16GB GPU 可跑）

---

### Q4: 如何把多模态大模型转化为Agent

**Agent = 大模型 + 工具 + 记忆 + 循环推理**

通过 Function Calling 实现。给大模型定义一组工具（control_light、control_ac 等），大模型根据看到的场景自主决定调用哪些工具、传什么参数。

核心循环：
```
OBSERVE(观察画面+传感器) → THINK(大模型推理) → ACT(执行工具调用) → REFLECT(记录+学习)
```

Agent 的 System Prompt 中包含角色定义、决策原则、可用工具列表。大模型会**自主决定**该做什么，而不是我们写 IF-ELSE。

## 二、AI能力类

### Q5: YOLO+InsightFace能否满足人脸检测

**可以，但只能解决"谁在哪里"的问题。**

- YOLO v8 做人体检测：准确率 mAP > 90%，告诉你"画面中有 3 个人"
- InsightFace 做人脸识别：1:N 准确率 > 99%，告诉你"这是张三"

**它能做到的：**
- 某个工位是否有人 ✓
- 这个人是谁 ✓
- 区域内有多少人 ✓

**它做不到的：**
- 这个人在干什么 ✗（需要多模态大模型）
- 这个人的情绪状态 ✗
- 场景的语义理解 ✗

结论：YOLO+InsightFace 作为"感知层"合适，但不能替代"认知层"。

---

### Q6: 人的行为检测怎么办

三种技术路线：

| 路线 | 技术 | 能力 | 是否需要训练 |
|------|------|------|------------|
| A. 传统模型 | MediaPipe/OpenPose 骨骼关键点 | 站立、坐下、举手等基础动作 | 需要针对办公场景训练 |
| B. 视频理解模型 | SlowFast / VideoMAE | 打电话、吃东西、写字、开会 | 需要大量标注数据 |
| C. 多模态大模型(推荐) | Qwen2.5-VL / GPT-4o | 几乎所有日常办公行为 | 无需训练，开箱即用 |

**推荐路线 C**：直接把视频帧发给多模态大模型，用自然语言问"这个人在做什么？"——能理解几乎所有日常办公行为，且无需专门训练。

### Q7: 摄像头实时推送视频帧的处理速度如何

不同模型处理速度对比：

| 模型类型 | 单帧延迟(GPU) | 单帧延迟(CPU) | 适合场景 |
|---------|-------------|-------------|---------|
| YOLO v8n | ~2ms | ~40ms | 实时检测 |
| InsightFace | ~15ms | ~100ms | 人脸识别 |
| 多模态大模型(本地7B) | ~1-3s | ~10-30s | 场景理解 |
| 多模态大模型(云端API) | ~2-5s | N/A | 场景理解 |

**关键结论：多模态大模型不可能逐帧处理，但也不需要。**

正确做法是**分层处理**：
- YOLO 每秒采样 1 帧做实时检测（够快）
- 大模型只在**事件触发**时处理（人员变化、传感器异常等），不是每帧都调

### Q8: 本地7B大模型性能够不够聪明，怎么做到实时无感

**7B 模型场景理解能力不错，但 Function Calling 准确率约 85-90%。**

复杂推理（如多人偏好协调）时 7B 可能给出不合理答案。建议：
- 日常简单场景(80%)：7B 本地处理 → 足够
- 复杂场景(15%)：降级调用云端大模型 → 能力更强
- Function Calling 出错时：重试 1 次，仍失败则用预设规则兜底

**四层策略实现"实时无感"**（用户感知 < 1 秒）：

| 策略 | 触发条件 | 延迟 | 原理 |
|------|---------|------|------|
| 1. 缓存命中 | 相同场景(同一人+同一房间)已有决策 | ~50ms | Redis 缓存上次决策，直接复用 |
| 2. 反射层默认动作 | 缓存未命中 | ~50ms | 先用默认亮度开灯，不等大模型 |
| 3. 认知层异步精调 | 反射层已响应后 | 2-5秒 | 大模型推理完成后静默微调灯光/空调到偏好值 |
| 4. 预测式预加载 | 飞书考勤+历史规律 | 提前准备 | 预测员工到达时间，提前开灯开空调 |

用户体验时间线：
```
0ms      张三走进办公室
50ms     灯亮了(反射层/缓存)        ← 用户感知：即时
2000ms   灯光平滑过渡到偏好值       ← 用户感知：自然，几乎无感
```

## 三、Agent核心机制类

### Q9: Agent框架详细说明与实现路径

**方案采用自研轻量 Agent Loop，不使用 LangChain。**

Agent 框架由 6 个组件构成：

| 组件 | 文件 | 职责 | 代码量 |
|------|------|------|-------|
| LLM客户端 | `agent/llm_client.py` | 封装本地Ollama和云端API的调用，支持自动降级和重试 | ~80行 |
| Agent主循环 | `agent/agent_loop.py` | Observe→Think→Act→Reflect 四步循环 | ~100行 |
| 工具定义 | `agent/tools.py` | 7个工具的 JSON Schema 声明(纯数据) | ~120行 |
| 工具执行器 | `agent/tool_executor.py` | 把大模型输出的JSON路由到真实函数执行 | ~150行 |
| Prompt构建器 | `agent/prompt_builder.py` | 动态拼装图像+文本+记忆为多模态Prompt | ~60行 |
| 冲突协调器 | `agent/conflict_resolver.py` | 多人偏好不一致时的协调 | ~50行 |

**为什么不用 LangChain：**

| 考量 | LangChain | 自研 Agent Loop |
|------|-----------|----------------|
| 代码量 | 引入数十个依赖包 | 6个文件，~500行代码 |
| 灵活性 | 被框架约束，IoT场景适配困难 | 完全按需定制 |
| 调试 | 黑盒，出错难追踪 | 每一步都透明可追踪 |
| 依赖风险 | 版本更新频繁，API常breaking change | 无外部框架依赖 |

本项目的 Agent 逻辑本身不复杂，复杂度在设备协议适配和记忆系统，这些 LangChain 帮不上忙。

### Q10: 工具执行器(Tool Executor)是什么，怎么定义

**工具执行器是连接大模型"想法"和真实世界"动作"的桥梁。**

大模型通过 Function Calling 输出的只是一段 JSON：
```json
{"name": "control_light", "arguments": {"room_id": "zone_a", "action": "on", "brightness": 70}}
```

工具执行器的职责：**解析 JSON → 找到对应函数 → 调 MQTT/HTTP/飞书 API → 返回结果**

每个 `_handle_xxx` 函数都是**薄封装**，调用现成的 Python 库：

| handler函数 | 实际做什么 | 用到的库 | 代码量 |
|------------|-----------|---------|-------|
| `_handle_light` | MQTT publish 灯光控制指令 | paho-mqtt | ~10行 |
| `_handle_ac` | MQTT publish 空调控制指令 | paho-mqtt | ~10行 |
| `_handle_screen` | MQTT publish 屏幕内容 | paho-mqtt | ~8行 |
| `_handle_fresh_air` | MQTT publish 新风档位 | paho-mqtt | ~5行 |
| `_handle_get_preference` | SQLite查询 + ChromaDB向量搜索 | sqlalchemy, chromadb | ~10行 |
| `_handle_notify` | HTTP POST 飞书发消息API | httpx | ~8行 |
| `_handle_update_memory` | ChromaDB add() 写入记忆 | chromadb | ~8行 |

**不需要写 AI 逻辑**（大模型处理决策），只需要写"怎么让灯亮起来"这个层面的设备通信代码。

### Q11: Agent如何通过员工喜好偏好实时调控硬件

完整流程：

```
1. 摄像头检测到张三进入A区
2. Agent 查询记忆系统 → 获取张三偏好(灯光暖光3500K 亮度70%, 空调25度)
3. Agent 检查A区还有谁 → 李四(偏好冷光5000K, 空调23度)
4. Agent 推理协商 → 灯光取折中4200K, 空调取24度
5. Agent 调用 control_light(brightness=70, color_temp=4200)
6. Agent 调用 control_ac(temperature=24)
```

**多人偏好冲突处理策略（交给大模型自主协商）：**
- 温度/亮度取在场人员偏好的折中值
- 离设备更近的人权重更高
- 把所有人的偏好告诉 Agent，让它自己推理最佳方案

## 四、记忆系统类

### Q12: 记忆系统怎么实现（具体步骤）

记忆系统分为四个阶段：初始化 → 写入 → 读取 → 学习。

**阶段一：初始化（系统启动时）**

| 步骤 | 操作 | 存储 |
|------|------|------|
| 1 | 创建 employees 表 + preferences 表 | SQLite |
| 2 | 创建 implicit_preferences 集合 + context_memories 集合 | ChromaDB |
| 3 | 启动 Redis | Redis |

**阶段二：写入记忆（三个来源）**

| 来源 | 触发场景 | 存到哪里 | 示例 |
|------|---------|---------|------|
| 员工主动设置 | 飞书小程序设置偏好 | SQLite preferences 表 | 张三: 空调25度 |
| Agent 观察学习 | 员工手动改了Agent的设置 | ChromaDB (向量+文本) | "张三把空调从25改为23" |
| 设备执行结果 | 工具执行器完成控制 | Redis (key-value) | device:ac_a1:state → {temp:23} |

**阶段三：读取记忆（Agent决策时）**

```
事件: 张三进入A区会议室
  ↓
步骤1: 查SQLite → {ac:25度, light:3500K}（显式偏好）
步骤2: 查ChromaDB → "张三开会时偏好22度冷光"（向量相似度搜索）
步骤3: 查Redis → 当前空调关着、灯关着（实时设备状态）
步骤4: 拼成文本注入Agent Prompt
```

**阶段四：学习（Agent反思时）**

```
Agent设了25度 → 10分钟后张三手动调到23度
  ↓
PreferenceLearner检测到差异
  ↓
生成记忆: "张三在[当前情境]下偏好23度而非25度"
  ↓
写入ChromaDB → 下次直接设23度
```

### Q13: 员工偏好如何被长期记忆保存并服务到每个人

**三层记忆架构：**

| 层级 | 存储 | 内容 | 来源 | 示例 |
|------|------|------|------|------|
| 第1层：显式偏好 | SQLite | 员工主动设置的偏好 | 飞书小程序 | 张三: 空调25度, 暖光3500K |
| 第2层：隐式偏好 | ChromaDB | Agent观察学习到的偏好 | PreferenceLearner | 张三实际偏好23度(连续3天手动调) |
| 第3层：情境记忆 | ChromaDB | 与具体场景绑定的偏好 | 带场景标签的观察 | 张三开会时22度冷光，独自办公25度暖光 |

Agent 决策时通过 RAG（检索增强生成）从三层记忆中拉取相关信息，组装为文本注入到 Prompt 中，大模型据此做出个性化决策。

## 五、硬件与网络类

### Q14: 摄像头有什么具体要求

**硬件参数：**

| 参数 | 最低要求 | 推荐 | 原因 |
|------|---------|------|------|
| 分辨率 | 1080P | 2K (2560x1440) | 人脸识别需面部 >= 80x80 像素 |
| 帧率 | 15fps | 25fps | 1fps 采样足够，高帧率有助于抓清晰帧 |
| 编码 | H.264 | H.265 | H.265 带宽减半 |
| 协议 | 必须 RTSP | RTSP + ONVIF | RTSP 是视频流接入标准 |
| 夜视 | IR 红外 | 星光级传感器 | 加班/暗光环境仍需检测 |
| 视角 | 90度 | 120度广角 | 覆盖更大区域，减少摄像头数量 |
| 供电 | - | PoE (802.3af) | 一根线解决供电+网络 |
| 网络 | WiFi | 有线以太网 | WiFi 不稳定会导致视频流中断 |

**安装位置策略：**

| 场景 | 摄像头类型 | 安装位置 | 目标 |
|------|-----------|---------|------|
| 大门入口 | 枪机(定焦) | 门口正上方2.5-3米，俯角15-20度 | 人脸正脸捕获 |
| 办公区域 | 半球(广角) | 天花板中央 | 区域人数统计 |
| 会议室 | 半球(广角) | 门口对角天花板 | 人员进出 |

**关键注意：** 人脸识别入口摄像头侧面角度不能超过 45 度，否则准确率急剧下降。

### Q15: 具体需要怎样的网络方案

**带宽需求：**
```
单路摄像头:
  主码流 1080P H.265 = 2~4 Mbps（录像用）
  副码流 720P（AI分析用）= 0.5~1 Mbps
  
10路摄像头总需求: ~50 Mbps → 千兆交换机即可
```

**最简网络方案（单个办公室 < 300 平方）：**

```
一台16口PoE千兆交换机(~800元)
  ├── 端口1-10:  摄像头(PoE供电+数据)
  ├── 端口11-12: 传感器网关
  ├── 端口13:    EaseAgent服务器
  ├── 端口14:    上联到办公室路由器
  └── 端口15-16: 预留

所有设备同网段 192.168.1.x，不需要 VLAN
网线: 六类(CAT6) 无氧铜，单段 ≤ 80 米
```

**推荐交换机型号：**
- 8口：海康 DS-3E0510P-E（~400元）
- 16口：TP-LINK TL-SG1218PE（~800元）

### Q17: OpenClaw能否加入这个环节

**结论：不建议作为核心框架，可作为可选的对话入口。**

| 维度 | OpenClaw的设计 | EaseAgent的需求 | 契合度 |
|------|---------------|-----------------|-------|
| 交互模式 | 聊天驱动(用户发消息→Agent回复) | 事件驱动(摄像头检测→Agent决策) | 低 |
| 输入类型 | 文本/图片消息 | 实时视频流+传感器数据 | 低 |
| 工具调用 | http_request/shell等通用工具 | MQTT设备控制等IoT专用工具 | 中 |
| 通信渠道 | WhatsApp/Telegram/Discord | 飞书 | 低(无原生飞书支持) |
| 部署模式 | 个人助手(单用户) | 办公室系统(多用户多设备) | 低 |

**不推荐的原因：**
1. 设计范式不匹配 — OpenClaw 是对话式 Agent，EaseAgent 是感知式 Agent
2. 实时性不足 — Gateway 架构有消息队列开销，不适合 50ms 反射响应
3. 增加不必要的复杂度

**唯一可取之处：** 如果未来想让员工在飞书里用自然语言控制办公室（"把灯调亮一点"），OpenClaw 可作为对话入口接入，但不替代核心 Agent。

### Q18: 围绕"落地"还有哪些必须解决的问题

方案文档之外必须解决的 7 个落地问题：

**1. 设备协议兼容性**

| 设备类型 | 常见协议 | 适配方案 |
|---------|---------|---------|
| WiFi智能灯 | 厂商私有API | tuya-connector SDK → 封装为 MQTT |
| Zigbee智能灯 | Zigbee 3.0 | Zigbee2MQTT 网关 → 自动桥接 |
| 分体空调 | 红外 | Broadlink RM4 红外转发器 → HTTP → MQTT |
| 中央空调 | BACnet/Modbus | 协议网关(Intesis) → HTTP/MQTT |
| 显示屏(安卓) | - | 安卓端运行 WebView 客户端，WebSocket 接收内容 |

**2. 故障降级**
- 大模型挂了 → 反射层+缓存继续工作
- MQTT 挂了 → watchdog 自动重启
- 服务器宕机 → 物理开关仍可手动控制
- 断网(混合模式) → 降级为反射层+缓存

**3. 隐私合规**
- 人脸识别涉及《个人信息保护法》敏感个人信息
- 需要员工书面/电子签署知情同意书
- 人脸特征向量加密存储
- 提供退出机制（不参与人脸识别的员工降级为仅人员检测）

**4. 网络基础设施**
- 摄像头与服务器同一网段/VLAN
- PoE 千兆交换机
- 视频流不抢占办公网络带宽

**5. 分阶段上线**
- 阶段1(影子模式): Agent只记录决策不控制设备，验证准确率 > 80%
- 阶段2(单房间试点): 选一个会议室开启实际控制
- 阶段3(逐步扩展): 每周增加一个区域
- 阶段4(全量运行): 持续优化

**6. 运维监控**
- 摄像头在线率、AI 推理延迟、设备心跳、Agent 决策日志
- watchdog 看门狗服务，异常自动重启

**7. 员工体验**
- 飞书小程序简单到一看就会
- 提供"一键关闭 AI 控制"的能力
- 初期屏幕显示 Agent 决策原因，增加透明度

## 七、技术概念类

### Q19: APScheduler / SQLite+Redis / Vue 3+Vite 是什么

**APScheduler（任务调度器）**

Python 的定时任务库，类似"闹钟管理器"。在本项目中负责：
- 空调预启动（"每天 8:30 开空调"）
- 定时巡检设备状态
- 定期清理过期日志

相比 Linux crontab，它更灵活——可以在 Python 代码中动态添加/修改/删除定时任务。

**SQLite + Redis（双数据库）**

| | SQLite | Redis |
|--|--------|-------|
| 存储位置 | 硬盘（持久化） | 内存（快速读写） |
| 比喻 | 档案柜 | 白板 |
| 存什么 | 员工信息、设备配置、偏好设置、历史记录 | 设备实时状态、决策缓存、厕位占用 |
| 特点 | 零配置、单文件、不需安装数据库服务 | 读写极快(毫秒级)，适合频繁读取 |

**Vue 3 + Vite（前端技术）**

- **Vue 3** = 前端 JavaScript 框架，把后端 API 数据渲染成好看的网页界面
- **Vite** = Vue 的开发构建工具，把代码编译打包成浏览器能运行的文件

类比关系：

| 后端 | 前端 |
|------|------|
| Python 是编程语言 | JavaScript 是编程语言 |
| FastAPI 是后端框架 | Vue 3 是前端框架 |
| pip 是包管理器 | npm 是包管理器 |
| uvicorn 启动服务 | Vite 启动开发服务 |

生产部署时，Vite 编译出静态文件放到 Nginx 中，用户浏览器访问即可看到管理面板。

### Q20: FastAPI 在项目中的作用

**FastAPI 不是"转发器"，而是后端应用服务器——所有外部请求都通过它进出，它在中间执行业务逻辑。**

在 EaseAgent 中具体承担三个职责：

**职责1：给前端提供 REST API（数据的增删改查）**

Vue 管理面板和飞书小程序不能直接操作数据库或控制设备，通过 HTTP 请求和 FastAPI 通信。FastAPI 收到请求后做数据校验、查询/写入数据库、通知内部组件，然后返回结果。

4 个真实场景示例：

| 场景 | 请求 | FastAPI 做了什么 |
|------|------|-----------------|
| 员工在飞书设置"我喜欢23度" | POST /preferences | 校验数据范围 → 查员工是否存在 → 写入数据库 → 通知事件总线 → 返回"成功" |
| 管理员网页添加新设备 | POST /devices | 校验设备类型 → 查重 → 存数据库 → 更新设备注册中心 → 订阅MQTT心跳 → 返回设备ID |
| 员工查厕所空位 | GET /toilet/status | 从Redis读实时厕位状态 → 格式化 → 返回JSON |
| 管理员查Agent决策日志 | GET /agent-logs | 按条件筛选数据库 → 返回决策记录列表(时间/触发事件/推理/动作/结果) |

**职责2：WebSocket 实时推送（服务器主动推消息给前端）**

REST API 是"前端问，后端答"，但厕位变化、设备掉线等需要服务器**主动推送**。WebSocket 是全双工长连接，有新数据时立刻推给客户端，不需要前端轮询。

**职责3：管理应用生命周期（启动/关闭时的初始化和清理）**

FastAPI 的 lifespan 在系统启动时自动连接 MQTT Broker、Redis、初始化数据库、启动事件总线、启动设备心跳检测；关闭时断开连接、清理资源。

**类比：** FastAPI = 餐厅服务员。不是简单传话，而是检查菜单（数据校验）、记账（操作数据库）、传菜通知厨房（调用MQTT/事件总线）、协调（业务逻辑）、主动上菜（WebSocket推送）。

### Q21: 系统内部消息路由机制

**"消息发给谁"在三个层面各有不同的路由机制：**

| 场景 | 谁负责路由 | 怎么找到对方 |
|------|----------|------------|
| 前端请求 API | FastAPI | URL 路径（`/devices`、`/toilet`），类似"分机号" |
| 服务器控制设备 | MQTT Broker | Topic 地址，类似"收件地址" |
| 服务器推送前端 | WebSocket | 客户端订阅的事件类型 |

**MQTT Topic 路由（最关键的部分）：**

Topic 是层级结构的字符串：`easeagent/zone_a/light/light_a1/cmd`（项目名/房间/设备类型/设备ID/指令）

每个设备上线时只订阅自己地址的 topic。Agent 发送控制指令时，MQTT Broker 根据 topic 精准投递：

```
Agent → publish("easeagent/zone_a/light/light_a1/cmd", {brightness:70})
                            ↓
MQTT Broker 查看谁订阅了这个 topic → 只有 light_a1 订阅了
                            ↓
light_a1 收到 → 调到70%
light_a2 / ac_a1 完全不受影响
```

批量控制用通配符：`easeagent/zone_a/light/+/cmd` 匹配所有 zone_a 的灯。

### Q22: Docker Compose 编排是什么

**Docker Compose 用一个 YAML 文件描述所有依赖服务，一条命令全部启动。**

EaseAgent 依赖 3 个外部服务：

| 服务 | 作用 | 如果没有它 |
|------|------|----------|
| Mosquitto | MQTT 消息中转站 | 服务器和设备之间无法通信 |
| Redis | 内存缓存 | 无法秒级响应，每次都要重新推理 |
| ChromaDB | 向量数据库 | Agent没有记忆，无法个性化 |

传统做法：在服务器上手动安装这三个软件，分别配置端口、数据目录、启动参数。换台服务器全部重来。

Docker Compose 做法：写一个 `docker-compose.yml` 描述所有服务，然后 `docker-compose up` 一条命令自动下载镜像、创建隔离容器、按依赖顺序启动、自动组网（容器之间可用服务名互相访问）。

**类比：** 没有 Docker Compose = 自己买灶台装冰箱接水管开餐厅，每开一家分店重新装修。有 Docker Compose = 写好装修图纸，任何空铺面喊一声"照着建"，全部自动到位。

### Q23: 事件总线是什么

**事件总线是系统内部的"广播喇叭"，解决模块之间的通信解耦问题。**

问题：摄像头检测到人进来了，灯光/空调/屏幕/Agent/日志模块都需要知道。

**没有事件总线（蜘蛛网式耦合）：**

摄像头模块必须认识所有其他模块，逐个调用。新增一个模块要改摄像头代码，删掉一个模块摄像头会崩。

**有事件总线（广播式解耦）：**

各模块启动时注册"我对某类事件感兴趣"。摄像头只管对着广播喇叭喊"有人进来了"，所有订阅了该事件的模块自动收到通知。新增模块只需注册一行，删掉模块零影响。

核心原理（3 行代码）：
```python
class EventBus:
    listeners = {}  # {"事件类型": [处理函数列表]}
    def subscribe(self, event_type, handler): ...   # 注册关注
    async def publish(self, event_type, data): ...  # 广播事件
```

EaseAgent 中的具体事件类型：

| 事件 | 谁发布 | 谁订阅 |
|------|-------|-------|
| `person_entered` | YOLO检测到人进入 | Agent、灯光反射层、日志 |
| `person_left` | YOLO检测到人离开 | Agent、灯光反射层（关灯） |
| `face_recognized` | InsightFace识别出身份 | Agent（查偏好）、屏幕（迎宾） |
| `sensor_update` | 传感器上报数据 | Agent、新风反射层 |
| `co2_high` | CO2超标 | 新风反射层（直接加大风量） |
| `device_offline` | 设备心跳超时 | 监控模块（报警） |
| `preference_updated` | 员工修改偏好 | Agent（下次用新偏好） |
| `agent_decision` | Agent做出决策 | 日志模块、WebSocket推送 |

**类比：** 没有事件总线 = 打电话（摄像头挨个打给灯光、空调、屏幕）。有事件总线 = 公司广播（摄像头喊一声，所有订阅频道的部门自动听到）。

## 八、Phase 1 设计逻辑类

### Q24: Phase 1 各步骤的原因和逻辑

**Phase 1 的定位是"打地基 + 搭框架"，不做任何 AI 功能，但构建后续所有 Phase 依赖的基础管道。**

Phase 1 构建了 6 套管道系统：

| 管道 | 对应文件 | 作用 | 后续谁用 |
|------|---------|------|---------|
| 配置管道 | `config/` + `core/config.py` | YAML配置 → Pydantic对象 → 全局可用 | 所有模块读取配置 |
| 数据管道 | `core/models.py` + `database.py` | ORM模型 → SQLite持久化 | 员工/设备/偏好/日志存储 |
| 消息管道 | `iot/mqtt_client.py` | 异步MQTT客户端 ↔ Mosquitto Broker ↔ IoT设备 | Agent控制设备 |
| 事件管道 | `core/event_bus.py` | 内部各模块解耦通信 | 反射层→认知层→记忆层 |
| API管道 | `api/routes/` + `websocket/` | REST + WebSocket对外接口 | Vue面板、飞书小程序 |
| 设备管道 | `iot/device_registry.py` + `protocols/` | 统一设备管理 + 协议适配 | 工具执行器控制设备 |

各步骤的具体原因：

**1. 根目录文件（工程规范）**

- `docker-compose.yml`：一键拉起Mosquitto/Redis/ChromaDB，不手动装
- `requirements.txt`：锁死Python依赖版本，防止环境不可复现
- `.env.example`：敏感信息（API Key）不写死在代码里
- `.gitignore`：防止数据库文件、密钥等被提交到git

**2. 配置文件（灵活性）**

配置与代码分离。不同环境（开发/生产）改yaml就行，不改代码。`agent_prompt.yaml` 可热更新提示词。

**3. 核心层（骨架）**

- `config.py`：Pydantic自动校验配置类型，支持环境变量覆盖
- `models.py` + `database.py`：所有业务数据持久化的基础
- `event_bus.py`：解耦的关键——后续Phase 2写反射层、Phase 3写认知层可以完全独立开发
- `main.py`：生命周期管理，启动时连接所有服务，关闭时清理资源
- `dependencies.py`：FastAPI依赖注入，测试时可mock

**4. IoT通信层（硬件桥梁）**

- `mqtt_client.py`：异步封装，支持自动重连/消息回调/通配符订阅
- `device_registry.py`：知道管理哪些设备、哪些在线
- `protocols/base.py`：统一接口基类，Phase 5填充具体协议

**5. API路由层（对外窗口）**

6个路由对应6个业务域（设备/房间/员工/偏好/日志/厕位），给Vue面板和飞书小程序提供数据接口。WebSocket提供实时推送能力。

**6. 辅助文件**

- `init_db.py`：首次运行创建数据库表
- `health_check.py`：检查各组件是否正常的骨架

**为什么要先做这些而不是直接写AI：** Phase 2 的反射层检测结果需要通过事件管道发出去；Phase 3 的认知层需要通过消息管道控制设备、通过数据管道查偏好。如果管道没搭好，AI 代码写了也没地方用。
